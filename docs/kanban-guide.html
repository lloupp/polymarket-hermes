HERMES AGENT

Modo Kanban

Sistema de tarefas distribuídas entre agentes especialistas — com dependências, paralelismo e human-in-the-loop.

01Quando Usar

Kanban brilha quando o trabalho é complexo, multi-agente, ou precisa sobreviver a crashes.

  • ✓
    Múltiplos especialistas — Pesquisa + análise + escrita requerem 3 perfis diferentes, cada um com seu contexto.
  • ✓
    Sobrevivência a crashes — Tarefas longas ou recorrentes que não podem perder progresso se o agente cair.
  • ✓
    Human-in-the-loop — O operador quer aprovar, revisar, ou intervir em qualquer passo do fluxo.
  • ✓
    Paralelismo — Fan-out: N tarefas independentes rodam ao mesmo tempo, fan-in no final.
  • ✓
    Revisão e iteração — Um reviewer bloqueia, o operador desbloqueia com feedback, o ciclo continua.
  • ✓
    Audit trail — Cada tarefa fica registrada em SQLite — histórico completo, sempre acessível.

🧩

Decompor & Criar

O orchestrador recebe seu pedido, quebra em subtarefas independentes e cria cada uma no board com kanban_create.

⚡

Dispatch Automático

O dispatcher pega tarefas ready e spawna o agente certo. Tarefas com parents esperam os pais terminarem.

🔄

Complete ou Block

Worker termina → kanban_complete(summary, metadata). Precisa de humano → kanban_block(reason).

🤝

Handoff

O summary + metadata do worker vira o input do próximo. Dependências se resolvem sozinhas.

03Roster de Especialistas

Perfis padrão que o dispatcher conhece. Cada um com workspace e ferramentas próprias.

researcher Lê fontes, coleta fatos, escreve achados
analyst Sintetiza, rankeia, deduplica múltiplos outputs de researcher
writer Escreve prosa na voz do usuário
reviewer Lê output, deixa findings, aprova ou bloqueia
backend-eng Escreve código server-side
frontend-eng Escreve código client-side
ops Rodar scripts, deploy, infra
pm Specs, critérios de aceite

kanban_create com parents 
# T1 e T2 rodam em paralelo — sem parents
t1 = kanban_create(
  title = "Pesquisar custos do Postgres",
  assignee = "researcher",
  body = "Compare custos de infra, migração e ops em 3 anos."
)

t2 = kanban_create(
  title = "Pesquisar performance do Postgres",
  assignee = "researcher",
  body = "Compare latência, throughput e scaling em ~500GB."
)

# T3 espera T1 e T2 completarem
t3 = kanban_create(
  title = "Sintetizar recomendação",
  assignee = "analyst",
  parents = [t1, t2],  # auto-promote quando ambos done
  body = "Leia findings de T1 e T2. Produza recomendação go/no-go."
)

# T4 espera T3 completar
t4 = kanban_create(
  title = "Escrever memo para CTO",
  assignee = "writer",
  parents = [t3],
  body = "Transforme recomendação em memo de 2 páginas."
)

06Padrões Comuns

4 padrões que cobrem 90% dos casos de uso.

Fan-out + Fan-in

PARALELO

N researchers trabalham em paralelo, 1 analyst sintetiza tudo no final.

R1 ──┐ R2 ──┼──→ A1 (parents: [R1, R2, R3]) R3 ──┘

Pipeline com Gate

SEQUENCIAL

Cada etapa depende da anterior. Reviewer pode bloquear e mandar de volta.

PM → Backend → Reviewer ──✓──→ Merge └──✗──→ Novo task p/ Backend (com feedback)

Fila Serial

BATCH

50 tarefas do mesmo tipo, sem dependências entre si. Dispatcher serializa por profile.

T1 → translator ──→ done T2 → translator ──→ done T3 → translator ──→ done ...

Human-in-the-Loop

INTERATIVO

Worker bloqueia com kanban_block(reason). Operador responde via /unblock.

Worker ──→ kanban_block("Qual API key usar?") ←── /unblock + comentário com a key Worker ──→ continua execução

07Exemplos Interativos

Clique nos tabs para ver diferentes cenários de uso.

Cenário: "Devemos migrar para Postgres?"

Orchestrator decompondo 
# Usuário pede: "Analise se devemos migrar para Postgres"

# 1. Fan-out: 2 researchers em paralelo
t1 = kanban_create(
  title = "Custos: Postgres vs atual",
  assignee = "researcher",
  body = "Compare infra, migração e ops em 3 anos. Fontes: AWS/GCP pricing, estimativas de tempo do time."
)

t2 = kanban_create(
  title = "Performance: Postgres vs atual",
  assignee = "researcher",
  body = "Compare latência, throughput e scaling em ~500GB, 10k QPS. Fontes: benchmarks, case studies."
)

# 2. Fan-in: analyst sintetiza
t3 = kanban_create(
  title = "Recomendação de migração",
  assignee = "analyst",
  parents = [t1, t2],
  body = "Leia findings de T1 e T2. Recomendação 1 página com trade-offs e go/no-go."
)

# 3. Writer transforma em memo
t4 = kanban_create(
  title = "Memo de decisão p/ CTO",
  assignee = "writer",
  parents = [t3],
  body = "Memo de 2 páginas baseado na recomendação do analyst."
)

# Resultado: T1 e T2 rodam em paralelo (~5min cada)
# T3 inicia automaticamente quando ambos completam
# T4 inicia quando T3 completa → memo final entregue

Cenário: "Implementar rate limiter"

Pipeline com review gate 
# Pipeline: PM → Eng → Reviewer → (aprova?) → Merge

t1 = kanban_create(
  title = "Spec: Rate limiter",
  assignee = "pm",
  body = "Especificar algoritmo (token bucket vs sliding window), chave (user_id vs IP), limits, config."
)

t2 = kanban_create(
  title = "Implementar rate limiter",
  assignee = "backend-eng",
  parents = [t1],
  body = "Implemente conforme spec do T1. Testes unitários + integração."
)

t3 = kanban_create(
  title = "Code review: rate limiter",
  assignee = "reviewer",
  parents = [t2],
  body = "Revise segurança (SQL injection, bypasses), performance, testes."
)

# Se reviewer bloquear:
# → kanban_block("SQL injection em /search, CSRF faltando")
# → Operador desbloqueia ou cria novo task p/ backend-eng corrigir
# → Reviewer roda de novo após correção

Cenário: "Deploy com validação humana"

Human-in-the-loop no deploy 
# Deploy com gate humano antes de ir pra produção

t1 = kanban_create(
  title = "Build + testes CI",
  assignee = "ops",
  body = "Rode build, testes, lint. Reporte resultado."
)

t2 = kanban_create(
  title = "Deploy staging",
  assignee = "ops",
  parents = [t1],
  body = "Deploy em staging. Rode smoke tests. Bloqueie para aprovação humana."
)

# Worker executa deploy em staging e bloqueia:
# kanban_block("Staging OK. Aprovar deploy para produção?")
# → Operador verifica staging e aprova via /unblock

t3 = kanban_create(
  title = "Deploy produção",
  assignee = "ops",
  parents = [t2],
  body = "Deploy em produção. Rollback automático se health check falhar."
)

08Referência CLI

Comandos do terminal para gerenciar o board manualmente.

hermes kanban create "título" --assignee researcher Criar tarefa
hermes kanban create "título" --assignee analyst --parent T1 Criar com dependência
hermes kanban show <id> Ver detalhes da tarefa
hermes kanban complete <id> --summary "feito" Completar tarefa
hermes kanban block <id> "preciso de X" Bloquear para aprovação humana
hermes kanban link --parent T1 --child T3 Criar dependência entre tarefas
hermes kanban tail <id> Acompanhar output em tempo real

09Handoff Entre Workers

O summary + metadata é o contrato entre agentes — shape importa.

kanban_complete(
  summary = "shipped rate limiter — token bucket, keys on user_id com IP fallback, 14 testes passando",
  metadata = {
    "changed_files": ["rate_limiter.py", "tests/test_rate_limiter.py"],
    "tests_run": 14,
    "tests_passed": 14,
    "decisions": ["user_id primary, IP fallback para unauthenticated"]
  }
)
kanban_complete(
  summary = "3 libs revisadas; vLLM wins em throughput, SGLang em latency, TRT-LLM em memória",
  metadata = {
    "sources_read": 12,
    "recommendation": "vLLM",
    "benchmarks": {"vllm": 1.0, "sglang": 0.87, "trtllm": 0.72}
  }
)
kanban_complete(
  summary = "reviewed PR #123; 2 issues bloqueantes (SQL injection, CSRF faltando)",
  metadata = {
    "pr_number": 123,
    "findings": [
      {"severity": "critical", "file": "api/search.py", "line": 42},
      {"severity": "high", "file": "api/settings.py"}
    ],
    "approved": False
  }
)
